# Deploy to Production — Полная церемония

> Этот файл выполняется Claude Code при запросе `deploy to production` / `задеплой в прод`.
> Каждый шаг обязателен. Пропускать шаги — нельзя.

---

## Шаг 1 — Сбор изменений с последнего прод-деплоя

```bash
git fetch origin
git log main..dev --oneline
git diff main..dev --stat
```

Выведи пользователю:
- Список коммитов (git log --oneline)
- Сводку изменённых файлов (--stat)
- Количество: N коммитов, X файлов изменено

---

## Шаг 2 — Health-check beta

```bash
curl -sf https://beta.dovstrechiapp.ru/health
```

Если beta unhealthy — **СТОП**. Сообщи: «Beta нездорова. Исправь проблемы на beta перед деплоем в prod.»
Продолжай только если beta вернула 200.

---

## Шаг 3 — Сгенерируй Changelog

На основе `git log main..dev --oneline` подготовь два формата:

**Формат A — технический** (для GitHub Release и `CHANGELOG.md`):
```
## [X.Y.Z] — DD.MM.YYYY

### Added
- ...

### Fixed
- ...

### Changed
- ...
```

**Формат B — для пользователей** (основа для Шага 3b):
```
## vX.Y.Z — DD.MM.YYYY

✨ ...
🐛 ...
⚡️ ...
```

Выведи оба формата пользователю.

---

## Шаг 3b — Предпросмотр: что увидят пользователи в приложении

На основе Формата B из Шага 3 подготовь финальный текст для экрана
«История обновлений» в приложении. Это то, что реальные пользователи
прочитают после обновления.

**Правила для пользовательского changelog:**
- Без технических терминов (не писать: «CORS», «HMAC», «asyncpg», «Docker», «nginx», «CI/CD», «VPS», «beta», «deploy», «worktree»)
- Инфраструктурные изменения (настройка деплоя, beta-окружение) — **не включать**
- Каждый пункт — одно законченное предложение от 5 до 15 слов
- Пиши от лица продукта: «Теперь можно...», «Исправлена...», «Добавлена...»
- Максимум 6 пунктов — выбери самые важные для пользователя
- Эмодзи в начале каждого пункта:
  - ✨ новая фича
  - 🐛 исправление бага
  - ⚡️ улучшение скорости/удобства
  - 🔒 безопасность (без деталей)
  - 🗓 изменения в логике расписаний/бронирований

**Выведи preview в точном формате как это будет выглядеть в приложении:**

```
╔══════════════════════════════════════════════╗
║   📱 PREVIEW: История обновлений в приложении ║
╠══════════════════════════════════════════════╣
║                                              ║
║  🆕 Версия X.Y.Z                             ║
║  DD месяц YYYY                               ║
║                                              ║
║  ✨ [пункт 1]                                ║
║  🐛 [пункт 2]                                ║
║  ⚡️ [пункт 3]                               ║
║  ...                                         ║
║                                              ║
╚══════════════════════════════════════════════╝
```

**После вывода preview — спроси:**

```
════════════════════════════════════════════════════════
  📝 Это увидят пользователи в разделе «Обновления»

  Хочешь отредактировать текст перед публикацией?

  Напиши:
  - ОК — текст устраивает, продолжаем
  - РЕДАКТИРОВАТЬ — и пришли исправленный текст
════════════════════════════════════════════════════════
```

Если пользователь пишет «РЕДАКТИРОВАТЬ» — жди новый текст от него,
используй его версию и снова покажи preview для финального подтверждения.

Если «ОК» — запомни финальный текст как `USER_CHANGELOG` и продолжай.

## ⛔ СТОП #1 — Подтверждение деплоя

Выведи итоговую сводку:
```
════════════════════════════════════════════════════════
  🚀 ДЕПЛОЙ В PRODUCTION

  Версия:    vX.Y.Z
  Изменений: N коммитов, X файлов
  Beta:      ✅ healthy

  [список ключевых изменений из Формата A]

  РЕАЛЬНЫЕ ПОЛЬЗОВАТЕЛИ БУДУТ ЗАТРОНУТЫ.
════════════════════════════════════════════════════════
```

Спроси первое подтверждение: **«Деплоить в production? (да/нет)»**

Подожди ответа. Если «нет» — СТОП, сообщи «Деплой отменён».

Если «да» — спроси второй раз: **«Это задеплоит на dovstrechiapp.ru (реальные пользователи). Подтверди ещё раз: да/нет»**

Если второе «да» — продолжай к Шагу 4.

## Шаг 4 — PR dev → main

```bash
git checkout dev
git pull origin dev
git checkout main
git pull origin main
git merge dev --no-edit
git push origin main
```

Выведи хеш последнего коммита после push.

---

## Шаг 5 — Деплой на production

GitHub Actions `deploy-prod.yml` запустится автоматически при push в main.

```bash
# Подожди 60–90 секунд, затем проверь статус workflow
gh run list --branch main --limit 1
```

Если gh недоступен — подожди 90 секунд и переходи к health-check.

---

## Шаг 6 — Health-check production

```bash
sleep 90
curl -sf https://dovstrechiapp.ru/health
```

Если unhealthy — немедленно сообщи пользователю. Не продолжай к Шагам 7–8 пока prod не healthy.

---

## Шаг 7 — GitHub Release

```bash
git tag -a "vX.Y.Z" -m "Release vX.Y.Z"
git push origin "vX.Y.Z"
```

Или через gh CLI:
```bash
gh release create "vX.Y.Z" \
  --title "vX.Y.Z — DD.MM.YYYY" \
  --notes "[технический changelog из Формата A]"
```

## Шаг 8 — Обновление CHANGELOG.md и CHANGELOG_USER.md

**Обнови `CHANGELOG.md`** — добавь новую версию после `## [Unreleased]`:

```bash
# Вручную или через sed — добавить запись из Формата A
```

**Также обнови `CHANGELOG_USER.md`** — добавь новую версию в начало файла:

```bash
python3 << 'PYEOF'
import datetime, re

version = "X.Y.Z"  # подставь реальную версию
user_changelog_text = """✨ ...
🐛 ...
⚡️ ..."""  # текст из Шага 3b (USER_CHANGELOG)

date_str = datetime.date.today().strftime('%d.%m.%Y')

new_entry = f"## v{version} — {date_str}\n\n{user_changelog_text}\n\n---\n\n"

with open('CHANGELOG_USER.md', 'r') as f:
    content = f.read()

insert_pos = content.find('\n---\n') + 5
updated = content[:insert_pos] + '\n' + new_entry + content[insert_pos:]

with open('CHANGELOG_USER.md', 'w') as f:
    f.write(updated)

print("✅ CHANGELOG_USER.md обновлён")
PYEOF

git add CHANGELOG.md CHANGELOG_USER.md
git commit -m "chore: update changelogs for vX.Y.Z"
git push origin main
```

## Схема церемонии

```
Шаг 1: Сбор изменений (git log main..dev)
Шаг 2: Health-check beta ✅
Шаг 3: Технический changelog (Форматы A и B)
Шаг 3b: ← Предпросмотр для пользователей + редактирование
⛔ СТОП #1: Двойное подтверждение деплоя
Шаг 4: Merge dev → main + push
Шаг 5: GitHub Actions deploy-prod.yml
Шаг 6: Health-check production ✅
Шаг 7: GitHub Release + тег
Шаг 8: CHANGELOG.md + CHANGELOG_USER.md
⛔ СТОП #2: Финальный отчёт
```
