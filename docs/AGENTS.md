# Правила для AI-агентов и Claude Code

Этот файл — обязательное чтение для любого AI-агента работающего с проектом.

## Правило #1: Beta-First Deployment

**НИКОГДА не деплой в production без явного запроса пользователя.**

Признаки явного запроса:
- Пользователь написал "задеплой в прод" / "deploy to production" / "релиз"
- Пользователь написал "в production" явно
- Пользователь прошёл процедуру подтверждения из `docs/DEPLOYMENT.md`

Признаки НЕ явного запроса (деплоить только на beta):
- "задеплой изменения"
- "примени фикс"
- "обнови сервер"
- "задеплой"
- любая неоднозначная формулировка

**При сомнении — спроси: "Деплоить на beta или production?"**

## Правило #2: Порядок деплоя

```
Всегда: dev → beta → (подтверждение) → main → production
Никогда: dev → main напрямую
```

## Правило #3: Перед деплоем в production

Обязательно выполни ДО начала деплоя:
1. `git diff main..dev --stat` — покажи что изменилось
2. `curl -sf https://beta.dovstrechiapp.ru/health` — beta должна быть healthy
3. Спроси подтверждение у пользователя
4. После "да" — спроси ещё раз: "Это задеплоит в PRODUCTION (реальные пользователи). Подтверди: да/нет"
5. Только после двух "да" — начинай деплой

## Правило #4: Ветки

| Ветка | Назначение | Деплой |
|-------|-----------|--------|
| `main` | Production-ready код | dovstrechiapp.ru |
| `dev` | Текущая разработка | beta.dovstrechiapp.ru |
| `feature/*` | Новые фичи | merge в dev, потом beta |
| `hotfix/*` | Срочные фиксы | merge в dev → beta → prod |

## Правило #5: Запрещено

- `git push origin main` напрямую из feature-ветки
- Деплой на prod без health-check beta
- Пропускать этапы подтверждения
- Деплоить если beta unhealthy
- Запускать `make deploy` без прохождения процедуры подтверждения
